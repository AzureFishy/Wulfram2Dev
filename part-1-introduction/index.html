    Wulfram 2 Server Implementation Guide¶

    Part 1: Introduction & Protocol Fundamentals¶

    Welcome¶

    This guide will walk you through implementing a Wulfram 2 server that successfully transitions clients from Login → Team Select → Game State. Each section builds on the previous one, with complete pseudo-examples you can use as templates.

    By the end of this guide, you'll understand: - How the Wulfram 2 protocol works at the byte level - What each packet does and why it's needed - How to construct valid packets with real example values - How to debug common issues

    Data Sources & Accuracy¶

    Important: This documentation was reconstructed through reverse engineering, as no original server source code or protocol specifications exist.

    Team Tactical Fan Site Game stats, build times, HP values (archived) Medium - player-tested

    Packet Captures Wire format verification High - when available

    Conventions used in this guide: - Values marked with ? are unknown/undocumented and need verification - Values marked as "Derived" were calculated or inferred from game behavior - Values marked as "Estimated" are educated guesses based on similar entities

    When implementing your server, start with the documented values and adjust based on testing.

    Packet Type Quick Reference¶

    The following packet types are used in the connection flow:

    Type Code Name Direction Size Part
    0x13 HELLO Server → Client Variable (3 sub-packets) Part 2
    0x11 LOGIN Client → Server Variable Part 3
    0x12 LOGIN_STATUS Server → Client ~4 bytes Part 3
    0x24 BEHAVIOR Server → Client ~3,118+ bytes Part 4
    0x25 TEAM_INFO Server → Client Variable Part 5
    0x30 REINCARNATE Client → Server ~5 bytes Part 6
    0x31 WORLD_STATS Server → Client Variable Part 7
    0x32 PLAYER Server → Client Variable Part 7
    0x33 UPDATE_ARRAY Server → Client Variable Part 7

    Note: The BEHAVIOR packet is by far the largest packet in the handshake sequence, containing all game configuration data including entity stats, weapon data, vehicle physics, and more.

    Fundamental Rule #1: Big-Endian Byte Order¶

    This is the most common source of bugs. Wulfram 2 uses BIG-ENDIAN byte order for all multi-byte values.

    What does Big-Endian mean?¶

    Big-Endian means the Most Significant Byte (MSB) comes first.

    Example: The number 0x4E89 (decimal 20105)

BIG-ENDIAN (Wulfram 2 uses this):
┌──────────┬──────────┐
│   0x4E   │   0x89   │   → Wire bytes: 4E 89
│  (high)  │  (low)   │
└──────────┴──────────┘
   Byte 0     Byte 1

LITTLE-ENDIAN (x86 native - DON'T USE):
┌──────────┬──────────┐
│   0x89   │   0x4E   │   → Wire bytes: 89 4E  ← WRONG!
│  (low)   │  (high)  │
└──────────┴──────────┘
   Byte 0     Byte 1

    How the Client Reads Big-Endian Data¶

    The client's read function shifts existing bits left and adds new bytes as the LSB:

    // From decompiled FUN_00508e70
uVar1 = uVar1 << 8 | (uint)*pbVar4;  // MSB arrives first

    And writes by extracting the MSB first:

    // From decompiled FUN_00508df0
*pbVar5 = (byte)(uVar6 >> 0x18);  // Write MSB (shift right 24 bits)
uVar6 = uVar6 << 8;               // Shift to get next byte

    Quick Reference: Big-Endian Encoding¶

    Value Type Big-Endian Bytes
    5 uint16 00 05
    2627 (0x0A43) uint16 0A 43
    20105 (0x4E89) uint32 00 00 4E 89
    1 uint32 00 00 00 01
    100 uint32 00 00 00 64

    Python Encoding¶

    import struct

# BIG-ENDIAN encoding (correct for Wulfram 2)
struct.pack('>H', 2627)      # Returns: b'\x0a\x43'  (2 bytes)
struct.pack('>I', 20105)     # Returns: b'\x00\x00\x4e\x89'  (4 bytes)
struct.pack('>i', 1)         # Returns: b'\x00\x00\x00\x01'  (4 bytes signed)

# The '>' character means BIG-ENDIAN
# 'H' = unsigned 16-bit (2 bytes)
# 'I' = unsigned 32-bit (4 bytes)
# 'i' = signed 32-bit (4 bytes)

    Fundamental Rule #2: Packet Frame Structure¶

    Every Wulfram 2 TCP packet has this structure:

    ┌──────────────┬──────────────────┬─────────────────────────────────┐
│    TYPE      │     LENGTH       │           PAYLOAD               │
│   (1 byte)   │  (2 bytes BE)    │      (LENGTH bytes)             │
├──────────────┼──────────────────┼─────────────────────────────────┤
│   0x13       │     00 05        │   00 00 00 4E 89                │
│   (HELLO)    │     (5)          │   (sub_type + version)          │
└──────────────┴──────────────────┴─────────────────────────────────┘
     ↑               ↑                      ↑
     │               │                      │
   Packet        Payload size          Actual data
    type         (NOT including       (this is what
                 type byte)           LENGTH counts)

    Important: The LENGTH field counts only the payload bytes, NOT the type byte!

    Implementation Note: The client's TCP receive handler operates as a state machine that reads the length field first, then processes the type and payload. Your server should send complete packets in the documented order: [TYPE:1][LENGTH:2][PAYLOAD:N]

    Example: HELLO Version Packet¶

    Complete packet: 13 00 05 00 00 00 4E 89
                 ── ───── ───────────────
                 │    │         │
                 │    │         └─ Payload: 5 bytes (sub_type + version)
                 │    └─ Length: 5 (0x0005 big-endian)
                 └─ Type: 0x13 (HELLO)

Payload breakdown:
                 00 00 00 4E 89
                 ── ───────────
                 │       │
                 │       └─ Version number: 0x00004E89 (20105) as uint32 BE
                 └─ Sub-type: 0x00 (version sub-packet)

Total wire size: 1 + 2 + 5 = 8 bytes

    Internal Bit-Stream Protocol¶

    Internally, Wulfram 2 uses a bit-oriented buffer system that can read arbitrary numbers of bits. However, the wire format uses byte-aligned frames. The internal read functions are:

    Function Bits Read Bytes Description
    FUN_005092e0 8 bits 1 byte Read uint8
    FUN_005092f0 16 bits 2 bytes Read uint16 (BE)
    FUN_00509310 32 bits 4 bytes Read uint32 (BE)
    FUN_00509340 variable variable Read length-prefixed string

    Fundamental Rule #3: Fixed-Point Number Encoding¶

    Wulfram 2 transmits floating-point values as fixed-point integers to save bandwidth.

    The Formula¶

    To ENCODE (float → wire):   wire_value = (int)(float_value × 65536)
To DECODE (wire → float):   float_value = wire_value × (1/65536)
                            float_value = wire_value × 0.0000152587890625

    Verification from Decompiled Code¶

    The client's decoding function confirms this exact formula:

    // From FUN_004dc620 (line 150538-150541)
float10 FUN_004dc620(int param_1) {
    return (float10)param_1 * (float10)1.52587890625e-05;
}
// Note: 1.52587890625e-05 = 1/65536 exactly

    Why 65536?¶

    65536 = 2^16. This gives 16 bits of fractional precision, which is enough for game physics.

    Examples¶

    Float Value Calculation Wire Value (decimal) Wire Value (hex BE)
    1.0 1.0 × 65536 = 65536 65536 00 01 00 00
    2.0 2.0 × 65536 = 131072 131072 00 02 00 00
    0.5 0.5 × 65536 = 32768 32768 00 00 80 00
    20.0 20.0 × 65536 = 1310720 1310720 00 14 00 00
    4.0 4.0 × 65536 = 262144 262144 00 04 00 00
    0.0 0.0 × 65536 = 0 0 00 00 00 00
    100.0 100.0 × 65536 = 6553600 6553600 00 64 00 00
    -1.0 -1.0 × 65536 = -65536 -65536 (signed) FF FF 00 00

    Python Encoding¶

    def encode_fixed_point(float_value):
    """Convert a float to Wulfram 2 fixed-point format (4 bytes big-endian)"""
    wire_value = int(float_value * 65536)
    return struct.pack('>i', wire_value)  # Use signed int for negative values

def decode_fixed_point(wire_bytes):
    """Convert Wulfram 2 fixed-point bytes back to float"""
    wire_value = struct.unpack('>i', wire_bytes)[0]
    return wire_value * (1.0 / 65536.0)

# Examples:
encode_fixed_point(1.0)   # Returns: b'\x00\x01\x00\x00'
encode_fixed_point(20.0)  # Returns: b'\x00\x14\x00\x00'
encode_fixed_point(0.5)   # Returns: b'\x00\x00\x80\x00'
encode_fixed_point(-1.0)  # Returns: b'\xff\xff\x00\x00'

    Fundamental Rule #4: Strings Are Length-Prefixed AND Null-Terminated¶

    Important: Strings in Wulfram 2 have TWO components: 1. A 16-bit big-endian length prefix (counts all bytes including the null terminator) 2. The string data, ending with a null byte (0x00)

    String: "Hello"

Wire format:
┌─────────────────┬──────────────────────────────┐
│  LENGTH         │  STRING DATA (with \0)       │
│  (2 bytes BE)   │  (LENGTH bytes)              │
├─────────────────┼──────────────────────────────┤
│    00 06        │  48 65 6C 6C 6F 00           │
│    (6)          │  H  e  l  l  o  \0           │
└─────────────────┴──────────────────────────────┘

Complete wire bytes: 00 06 48 65 6C 6C 6F 00
                     ───── ─────────────────
                       │         │
                       │         └─ "Hello" + null terminator (6 bytes)
                       └─ Length: 6 (includes the null terminator!)

    Verification from Decompiled Code¶

    Reading strings (FUN_00509340):

FUN_005094b0(uVar7);                     // Write 16-bit length prefix
// Then write string data...

    Python Encoding¶

    def encode_string(text):
    """
    Encode a string for Wulfram 2 protocol.
    Format: [length:2 bytes BE][string data with null terminator]
    """
    string_bytes = text.encode('utf-8') + b'\x00'  # Add null terminator
    length = struct.pack('>H', len(string_bytes))   # Length includes the \0
    return length + string_bytes

def decode_string(data, offset=0):
    """
    Decode a Wulfram 2 string from bytes.
    Returns (string, new_offset)
    """
    length = struct.unpack('>H', data[offset:offset+2])[0]
    string_data = data[offset+2:offset+2+length]
    # Remove null terminator for Python string
    text = string_data.rstrip(b'\x00').decode('utf-8')
    return text, offset + 2 + length

# Examples:
encode_string("Hello")        # b'\x00\x06Hello\x00'
encode_string("Red Team")     # b'\x00\x09Red Team\x00'
encode_string("127.0.0.1")    # b'\x00\x0a127.0.0.1\x00'
encode_string("")             # b'\x00\x01\x00'  (length 1 for just the null byte)

    Fundamental Rule #5: Booleans¶

    Boolean values are encoded as a single byte: - 0x00 = false - Any non-zero value = true (typically 0x01)

    def encode_bool(value):
    """Encode a boolean for Wulfram 2 protocol"""
    return bytes([0x01 if value else 0x00])

def decode_bool(byte_value):
    """Decode a Wulfram 2 boolean"""
    return byte_value != 0x00

    Summary: The Building Blocks¶

    Before we dive into specific packets, make sure you understand these fundamentals:

    Concept Rule Wire Format
    Byte Order Always BIG-ENDIAN for multi-byte integers MSB first
    Packet Frame [type:1][length:2 BE][payload:length] 3-byte header + payload
    Length Field Counts payload bytes only (excludes type byte) uint16 BE
    Floats/Doubles Encoded as int(value × 65536) int32 BE (signed)
    Strings Length-prefixed AND null-terminated [len:2 BE][data + \0]
    Booleans 1 byte: 0x00 = false, else = true uint8

    Quick Reference: Data Type Encoding¶

    import struct

# === INTEGERS ===
def encode_uint8(value):
    return bytes([value & 0xFF])

def encode_uint16(value):
    return struct.pack('>H', value)

def encode_uint32(value):
    return struct.pack('>I', value)

def encode_int32(value):
    return struct.pack('>i', value)

# === FIXED-POINT FLOATS ===
def encode_fixed_point(float_value):
    return struct.pack('>i', int(float_value * 65536))

# === STRINGS ===
def encode_string(text):
    string_bytes = text.encode('utf-8') + b'\x00'
    return struct.pack('>H', len(string_bytes)) + string_bytes

# === BOOLEANS ===
def encode_bool(value):
    return bytes([0x01 if value else 0x00])

# === PACKET WRAPPER ===
def make_packet(packet_type, payload):
    """Wrap payload in a complete packet frame"""
    return bytes([packet_type]) + struct.pack('>H', len(payload)) + payload
