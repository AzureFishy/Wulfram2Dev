    Wulfram 2 Server Implementation Guide¶

    Part 4-6: BEHAVIOR Packet - Packet Finalization & Byte Alignment¶

    Overview¶

    After all five data sections of the BEHAVIOR packet have been read, the client performs a byte alignment step to ensure the internal bit stream is properly synchronized. While this step is typically transparent for server implementations, understanding it is crucial for debugging protocol mismatches.

    Position in BEHAVIOR Packet:
┌─────────────────────────────────────────────────────────────────────────┐
│  SECTION 1: Fixed Header (95 bytes)                        ✓ Complete  │
│  SECTION 2: Entity Weapon Data (2,340 bytes)               ✓ Complete  │
│  SECTION 3: Entity Core Stats (468 bytes)                  ✓ Complete  │
│  SECTION 4: Vehicle Type Data (72 bytes)                   ✓ Complete  │
│  SECTION 5: Physics Tables + Vehicle Models (140+ bytes)   ✓ Complete  │
├─────────────────────────────────────────────────────────────────────────┤
│  PACKET FINALIZATION: Byte Alignment (0-7 bits)            ◄ YOU ARE   │
│                       Ensures bit stream is byte-aligned       HERE    │
└─────────────────────────────────────────────────────────────────────────┘

    Why Byte Alignment Exists¶

    Wulfram 2 internally uses a bit-oriented buffer system that can read and write arbitrary numbers of bits. While the BEHAVIOR packet sends byte-aligned data (8, 16, 32 bits), the internal bit counter may become misaligned if:

    1. Any sub-byte reads occurred (e.g., boolean flags)
    2. Protocol errors caused partial reads
    3. The packet structure changed between client/server versions

    The byte alignment step acts as a synchronization checkpoint that: - Verifies the packet was read correctly - Discards any residual bits before the next packet - Provides error detection for protocol mismatches

    FUN_00509410: Alignment Implementation¶

    From FUN_00509410 (lines 184780-184792):

    void __fastcall FUN_00509410(int param_1) {
    int iVar1;

    // Check if there's data remaining in the bit stream
    if (**(int **)(param_1 + 0x18) != 0) {

        // Get current bit buffer context
        iVar1 = FUN_004dc7d0(0);

        // Check if bit position is non-zero (not byte-aligned)
        if (*(int *)(iVar1 + 8) != 0) {

            // Read padding bits to reach byte boundary
            // Formula: 8 - current_bit_position
            FUN_00508ba0(param_1, iVar1, 0, 8 - *(int *)(iVar1 + 8));
        }
    }
    return;
}

    The Bit Stream Context Structure¶

    The client maintains a bit buffer context for reading packet data:

    Bit Buffer Context Structure:
┌─────────┬──────┬─────────────────────────────────────────────────────────┐
│ Offset  │ Size │ Description                                             │
├─────────┼──────┼─────────────────────────────────────────────────────────┤
│ 0x00    │ 4    │ Pointer to raw data buffer                              │
│ 0x04    │ 4    │ Current byte position                                   │
│ 0x08    │ 4    │ Current bit position within byte (0-7)                  │
│ 0x0C    │ 4    │ Byte offset (for multi-buffer scenarios)                │
│ 0x10    │ 4    │ Bit offset within current byte                          │
│ 0x14    │ 4    │ Total bits read/written                                 │
│ 0x18    │ 4    │ Total bits available in buffer                          │
└─────────┴──────┴─────────────────────────────────────────────────────────┘

    The key field is offset 0x08 (bit position within byte): - If 0: Already byte-aligned, no padding needed - If 1-7: Must read 8 - bit_position padding bits

    How Alignment Works¶

    Byte Alignment Process:
═══════════════════════════════════════════════════════════════════════════

Case 1: Already Aligned (bit_position = 0)
┌─────────────────────────────────────────────────────────────────────────┐
│  Last byte read:    [████████]                                         │
│  Bit position:               ^ = 0 (aligned)                           │
│  Action:            No padding needed                                  │
└─────────────────────────────────────────────────────────────────────────┘

Case 2: Not Aligned (bit_position = 3)
┌─────────────────────────────────────────────────────────────────────────┐
│  Last byte read:    [███.....] [next packet...]                        │
│  Bit position:         ^ = 3 (not aligned)                             │
│  Padding bits:      8 - 3 = 5 bits                                     │
│  Action:            Read 5 padding bits (value ignored)                │
│  After alignment:   [████████] [next packet...]                        │
│  Bit position:               ^ = 0 (now aligned)                       │
└─────────────────────────────────────────────────────────────────────────┘

Case 3: Not Aligned (bit_position = 7)
┌─────────────────────────────────────────────────────────────────────────┐
│  Last byte read:    [███████.] [next packet...]                        │
│  Bit position:              ^ = 7 (not aligned)                        │
│  Padding bits:      8 - 7 = 1 bit                                      │
│  Action:            Read 1 padding bit (value ignored)                 │
│  After alignment:   [████████] [next packet...]                        │
│  Bit position:               ^ = 0 (now aligned)                       │
└─────────────────────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════════════════

    FUN_00508ba0: The Bit Skip Function¶

    From FUN_00508ba0 (lines 184269-184292):

    void FUN_00508ba0(int param_1, undefined4 *param_2, int param_3, int param_4) {
    int iVar1;

    // Calculate total bits to skip: (param_3 * 8) + param_4
    param_4 = param_4 + param_3 * 8;

    // Calculate byte advancement
    iVar1 = param_2[2] + param_4 >> 3;

    // Update byte position
    param_2[1] = param_2[1] + iVar1;

    // Update bit position within byte (modulo 8)
    param_2[2] = param_2[2] + param_4 + iVar1 * -8;

    // Update bits consumed counter
    param_2[6] = param_2[6] + param_4;

    // Update remaining bits
    *(int *)(param_1 + 0x1c) = *(int *)(param_1 + 0x1c) - param_4;

    // Check if buffer is exhausted
    if (param_2[6] == param_2[5]) {
        // Free the buffer if fully consumed
        // ...
    }
    return;
}

    When called from FUN_00509410, the parameters are: - param_3 = 0 (no full bytes) - param_4 = 8 - bit_position (remaining bits in current byte)

    For Server Implementers: What This Means¶

    Good News: Usually Nothing to Do¶

    If your server sends byte-aligned data (which it should), the client's bit position will always be 0 at the end of BEHAVIOR processing, and no padding is needed.

    All standard Wulfram 2 data types are byte-aligned: - uint8: 8 bits (1 byte) - uint16: 16 bits (2 bytes) - uint32: 32 bits (4 bytes) - fixed-point: 32 bits (4 bytes) - Strings: Length prefix (2 bytes) + data (N bytes)

    When Alignment Matters¶

    Alignment becomes relevant when:

    1. Debugging Protocol Mismatches: If your packet is the wrong size, the client may end up with a non-zero bit position, and the alignment step will consume bits from the next packet.

    2. Custom Extensions: If you ever add sub-byte fields (unlikely), you'd need to add explicit padding.

    3. Error Detection: A non-zero bit position at alignment time could indicate a structural error in your packet.

    Protocol Mismatch Detection¶

    The client's bit reading function (FUN_00509050) includes error detection:

    // From FUN_00509050 (lines 184570-184575)
if (**(int **)(param_1 + 0x18) == 0) {
    _sprintf(&DAT_005d5f10,
             "Mem_buff::retrieve_bits out of hunks while reading [%s]",
             DAT_005d5f08);  // Contains "BEHAVIOR"
    FUN_0050a860(4, &DAT_005d5f10,
                 "This is most likely a protocol missmatch",
                 *(undefined4 *)(param_1 + 4));
    return 0;
}

    This error means: The client tried to read more data than the packet contains.

    Common causes: 1. Wrong section sizes: Missing or undersized sections 2. Wrong field counts: Incorrect number of entities/weapons/vehicles 3. Wrong data types: Using 2-byte values where 4-byte expected 4. Byte order errors: Little-endian instead of big-endian

    Server Implementation: No Explicit Alignment Needed¶

    For your server, you do NOT need to add any alignment bytes or padding to the BEHAVIOR packet. The standard sections are all byte-aligned:

    def create_behavior_packet(section_1, section_2, section_3, section_4, section_5):
    """
    Assemble the complete BEHAVIOR packet.

    All sections are byte-aligned - no padding needed!
    """
    payload = bytearray()

    payload.extend(section_1)   # 95 bytes (byte-aligned)
    payload.extend(section_2)   # 2340 bytes (byte-aligned)
    payload.extend(section_3)   # 468 bytes (byte-aligned)
    payload.extend(section_4)   # 72 bytes (byte-aligned)
    payload.extend(section_5)   # 140+ bytes (byte-aligned)

    # NO PADDING NEEDED - total is already byte-aligned

    # Create packet frame
    packet = bytearray()
    packet.append(0x24)  # BEHAVIOR packet type
    packet.extend(struct.pack('>H', len(payload)))  # Length
    packet.extend(payload)

    return bytes(packet)

    Debugging Alignment Issues¶

    If you encounter alignment-related errors, check:

    1. Total Packet Size¶

    def verify_behavior_packet_size(packet):
    """Verify BEHAVIOR packet has correct minimum size."""
    payload_length = struct.unpack('>H', packet[1:3])[0]

    expected_minimum = 95 + 2340 + 468 + 72 + 140  # = 3115 bytes

    if payload_length < expected_minimum:
        print(f"ERROR: Payload {payload_length} bytes, expected >= {expected_minimum}")

        # Calculate where the shortfall is
        remaining = payload_length
        sections = [
            ("Fixed Header", 95),
            ("Entity Weapon Data", 2340),
            ("Entity Core Stats", 468),
            ("Vehicle Type Data", 72),
            ("Physics + Models", 140),
        ]

        for name, size in sections:
            if remaining >= size:
                remaining -= size
                print(f"  ✓ {name}: {size} bytes OK")
            else:
                print(f"  ✗ {name}: Expected {size} bytes, only {remaining} remaining!")
                break
    else:
        print(f"OK: Payload {payload_length} bytes (minimum {expected_minimum})")

    2. Section Boundaries¶

    Verify each section starts at the correct offset:

    Section Offsets (within payload):
┌─────────────────────────────────────────────────────────────────────────┐
│  Section 1 (Fixed Header):       Offset 0      (95 bytes)              │
│  Section 2 (Entity Weapon):      Offset 95     (2340 bytes)            │
│  Section 3 (Entity Core Stats):  Offset 2435   (468 bytes)             │
│  Section 4 (Vehicle Type):       Offset 2903   (72 bytes)              │
│  Section 5 (Physics + Models):   Offset 2975   (140+ bytes)            │
│  End of minimum packet:          Offset 3115                           │
└─────────────────────────────────────────────────────────────────────────┘

    3. Byte Order¶

    Verify all multi-byte values use big-endian encoding:

    # CORRECT
struct.pack('>I', value)  # Big-endian (network byte order)

# WRONG
struct.pack('<I', value)  # Little-endian (x86 native)
struct.pack('I', value)   # Platform-dependent (BAD!)

    Summary: Byte Alignment¶

    Aspect Details
    Handler Function FUN_00509dc0 → FUN_00509410
    Purpose Align bit stream to byte boundary
    Padding Bits 0-7 bits (calculated as 8 - current_bit_position)
    Server Action None required - all sections are byte-aligned
    When It Matters Debugging protocol mismatches
    Error Indicator Non-zero bit position suggests structural issues

    What Happens Next?¶

    After successfully processing the BEHAVIOR packet:

    1. The client's bit stream is byte-aligned
    2. The client has received all game configuration data
    3. The client is ready for the next packet (TEAM_INFO)
    4. The game can proceed to team selection
