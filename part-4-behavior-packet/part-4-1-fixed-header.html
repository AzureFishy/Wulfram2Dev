    Wulfram 2 Server Implementation Guide¶

    Part 4-1: BEHAVIOR Packet - Overview & Fixed Header¶

    Overview¶

    The BEHAVIOR packet (0x24) is the largest and most complex packet in the Wulfram 2 protocol. It configures everything the client needs to know about the game: physics, weapons, vehicles, entity types, and more.

    This is also where most implementation bugs occur, so pay close attention!

    Client State After LOGIN_STATUS (status=8):
┌─────────────────────────────────────────┐
│  "I'm logged in successfully!"          │
│  "Now I need game configuration..."     │
│  "Waiting for BEHAVIOR packet..."       │
└─────────────────────────────────────────┘
          │
          ▼
Server must send BEHAVIOR (0x24) next!

    Why BEHAVIOR Is Challenging¶

    The BEHAVIOR packet has five data sections that must be sent in exact order, followed by packet finalization:

    BEHAVIOR Packet Structure:
┌─────────────────────────────────────────────────────────────────────────┐
│                         BEHAVIOR (0x24)                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │ SECTION 1: Fixed Header                           (95 bytes)     │  │
│  │ Global game settings, physics constants, timers                  │  │
│  │ Handler: Inline in FUN_0046dc00                                  │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              │                                         │
│                              ▼                                         │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │ SECTION 2: Entity Weapon Data                     (variable)     │  │
│  │ Weapon configs for each entity type × 13 slots                   │  │
│  │ Handler: FUN_004e2f90                                            │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              │                                         │
│                              ▼                                         │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │ SECTION 3: Entity Core Stats                      (468 bytes)    │  │
│  │ ALWAYS 39 entities × 12 bytes (build time, cost, HP)             │  │
│  │ Handler: FUN_004e4a80                                            │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              │                                         │
│                              ▼                                         │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │ SECTION 4: Vehicle Type Data                      (variable)     │  │
│  │ Physics params for each vehicle type × 36 bytes                  │  │
│  │ Handler: FUN_004e5ce0                                            │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              │                                         │
│                              ▼                                         │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │ SECTION 5: Physics Tables + Vehicle Models        (variable)     │  │
│  │ Physics lookup tables + Tank/Medic/Bomber behaviors              │  │
│  │ Handler: FUN_004e9630                                            │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              │                                         │
│                              ▼                                         │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │ PACKET FINALIZATION: Byte Alignment               (0-7 bits)     │  │
│  │ Ensures bit stream is byte-aligned before packet ends            │  │
│  │ Handler: FUN_00509dc0 → FUN_00509410                             │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

    Note: The "Byte Alignment" step is NOT a data section—it's packet finalization that pads any remaining bits to reach a byte boundary. It reads 0-7 padding bits (not bytes).

    The Handler Code¶

    Here's the complete client handler for BEHAVIOR (from FUN_0046dc00, lines 77941-77984):

    void FUN_0046dc00(param_1, param_2, param_3) {
    FUN_004e5080();                    // Initialize context
    FUN_00509db0("BEHAVIOR", param_3); // Set packet name (for error messages)
    FUN_004e5090(param_3);             // Set context pointer

    // ═══════════════════════════════════════════════════════════
    // SECTION 1: Read Fixed Header (95 bytes)
    // ═══════════════════════════════════════════════════════════

    // --- Byte 0: Main behavior flag ---
    FUN_005092e0(local_5);             // 1 byte:  main flag
    DAT_0067916d = local_5[0] != '\0';

    // --- Bytes 1-20: Physics doubles 1-5 (timing/physics params) ---
    FUN_004e51d0(&DAT_00679170);       // 4 bytes: round_time_minutes (×1000 for ms)
    FUN_004e51d0(&DAT_006792f8);       // 4 bytes: physics param 2
    FUN_004e51d0(&DAT_00679300);       // 4 bytes: physics param 3
    FUN_004e51d0(&DAT_00679308);       // 4 bytes: physics param 4
    FUN_004e51d0(&DAT_00679310);       // 4 bytes: physics param 5

    // --- Bytes 21-32: Physics integers 1-3 ---
    FUN_00509310(&DAT_0067917c);       // 4 bytes: physics int 1 (game limit)
    FUN_00509310(&DAT_006791b0);       // 4 bytes: physics int 2
    FUN_00509310(&DAT_006791b4);       // 4 bytes: physics int 3

    // --- Bytes 33-36: Gravity constant ---
    FUN_004e51d0(&DAT_005738b8);       // 4 bytes: gravity (used in projectile physics)

    // --- Bytes 37-44: Physics integers 4-5 ---
    FUN_00509310(&DAT_006791b8);       // 4 bytes: sky_height (map vertical bounds)
    FUN_00509310(&DAT_006791bc);       // 4 bytes: physics int 5

    // --- Bytes 45-48: Physics double 7 ---
    FUN_004e51d0(&DAT_006792a0);       // 4 bytes: physics threshold value

    // --- Bytes 49-92: Render floats 1-11 (camera/view parameters) ---
    FUN_004e5170(&DAT_00679180);       // 4 bytes: render param 1
    FUN_004e5170(&DAT_00679184);       // 4 bytes: render param 2
    FUN_004e5170(&DAT_00679188);       // 4 bytes: render param 3
    FUN_004e5170(&DAT_0067918c);       // 4 bytes: render param 4
    FUN_004e5170(&DAT_00679190);       // 4 bytes: render param 5
    FUN_004e5170(&DAT_00679194);       // 4 bytes: render param 6
    FUN_004e5170(&DAT_00679198);       // 4 bytes: render param 7
    FUN_004e5170(&DAT_0067919c);       // 4 bytes: render param 8
    FUN_004e5170(&DAT_006791a0);       // 4 bytes: render param 9
    FUN_004e5170(&DAT_006791a4);       // 4 bytes: render param 10
    FUN_004e5170(&DAT_006791ac);       // 4 bytes: render param 11 (range threshold)

    // --- Bytes 93-94: Game mode flags ---
    FUN_005092e0(local_5);             // 1 byte: friendly fire flag
    DAT_006791c0 = local_5[0] != '\0';
    FUN_005092e0(local_5);             // 1 byte: final flag
    DAT_006792c4 = local_5[0] != '\0';

    // ═══════════════════════════════════════════════════════════
    // SECTIONS 2-5: Call sub-handlers
    // ═══════════════════════════════════════════════════════════
    FUN_004e2f90(param_3);   // Section 2: Entity weapon data
    FUN_004e4a80(param_3);   // Section 3: Entity core stats
    FUN_004e5ce0(param_3);   // Section 4: Vehicle type data
    FUN_004e9630(param_3);   // Section 5: Physics + vehicle models

    // ═══════════════════════════════════════════════════════════
    // PACKET FINALIZATION: Byte alignment
    // ═══════════════════════════════════════════════════════════
    FUN_00509dc0(param_3, param_2);    // Align bit stream to byte boundary
}

    Understanding the Reader Functions¶

    The BEHAVIOR packet uses several reader functions. Here's what each one does:

    Function Bits Read Wire Bytes Target Type Description
    FUN_005092e0 8 bits 1 byte uint8 Reads a single byte (for booleans/flags)
    FUN_00509310 32 bits 4 bytes uint32 Reads a 32-bit integer (big-endian)
    FUN_004e5170 32 bits 4 bytes float Reads fixed-point → converts to float
    FUN_004e51d0 32 bits 4 bytes double Reads fixed-point → converts to double

    Important: Both FUN_004e5170 (float) and FUN_004e51d0 (double) read 4 bytes from the wire! The difference is only in the C type they store the result into. On the wire, both are 4-byte fixed-point values.

    Verification: Fixed-Point Conversion¶

    Both functions internally call FUN_00509310 to read 4 bytes, then convert using FUN_004dc620:

    // FUN_004e5170 (float reader) - line 157101
void FUN_004e5170(float *param_1) {
    FUN_00509310(&local_4);                    // Read 4 bytes
    fVar1 = (float10)FUN_004dc620(local_4);    // Convert from fixed-point
    *param_1 = (float)fVar1;
}

// FUN_004e51d0 (double reader) - line 157125
void FUN_004e51d0(double *param_1) {
    FUN_00509310(&local_4);                    // Read 4 bytes
    fVar1 = (float10)FUN_004dc620(local_4);    // Convert from fixed-point
    *param_1 = (double)fVar1;
}

// FUN_004dc620 (fixed-point decoder) - line 150538
float10 FUN_004dc620(int param_1) {
    return (float10)param_1 * (float10)1.52587890625e-05;  // = 1/65536
}

    Field Details from Decompiled Code¶

    Round Time (Offset 1, DAT_00679170)¶

    The first physics double is used as a round time in minutes. From line 23370-23378:

    // If round time is negative, default to 60 minutes
if (_DAT_00679170 < 0.0 != (_DAT_00679170 == 0.0)) {
    _DAT_00679170 = 60.0;
}
// Calculate end time: current_time + (round_time_minutes × 1000)
local_2c = (undefined4)(longlong)ROUND(dVar1 + _DAT_00679170 * 1000.0);

    Recommended value: 60.0 (60 minutes per round)

    Gravity Constant (Offset 33, DAT_005738b8)¶

    Used in projectile and entity physics. From lines 85215-85227:

    // Gravity applied with damping factor
fVar7 = (float10)(float)((float10)_DAT_005738b8 * (float10)-0.85 * fVar6);
*(float *)((int)_Memory + 0x1c) = (float)(fVar7 + (float10)*(float *)((int)_Memory + 0x1c));

    Recommended value: 20.0 (typical game gravity)

    Friendly Fire Flag (Offset 93, DAT_006791c0)¶

    Controls whether teammates can damage each other. From line 75309:

    // If same team AND friendly fire is disabled, don't apply damage
if ((*(int *)(local_14 + 0xf0) == *(int *)(unaff_ESI + 0xf0)) && (DAT_006791c0 == '\0')) {
    bVar1 = true;  // Same team, no friendly fire
}

    Values: 0x00 = friendly fire OFF, 0x01 = friendly fire ON

    Building the Fixed Header in Python¶

    import struct

def encode_fixed_point(float_value):
    """Convert a float to Wulfram 2 fixed-point format (4 bytes big-endian)"""
    wire_value = int(float_value * 65536)
    return struct.pack('>i', wire_value)

def create_behavior_fixed_header():
    """
    Create the fixed header section of the BEHAVIOR packet.
    Returns exactly 95 bytes.
    """
    data = bytearray()

    # ═══════════════════════════════════════════════════════════
    # Byte 0: Main behavior flag
    # ═══════════════════════════════════════════════════════════
    main_flag = True
    data.append(0x01 if main_flag else 0x00)
    # Current size: 1 byte

    # ═══════════════════════════════════════════════════════════
    # Bytes 1-20: Five physics doubles (fixed-point)
    # ═══════════════════════════════════════════════════════════
    round_time_minutes = 60.0  # 60 minute rounds (default)
    physics_double_2 = 1.0
    physics_double_3 = 1.0
    physics_double_4 = 1.0
    physics_double_5 = 1.0

    data.extend(encode_fixed_point(round_time_minutes))
    data.extend(encode_fixed_point(physics_double_2))
    data.extend(encode_fixed_point(physics_double_3))
    data.extend(encode_fixed_point(physics_double_4))
    data.extend(encode_fixed_point(physics_double_5))
    # Current size: 1 + 20 = 21 bytes

    # ═══════════════════════════════════════════════════════════
    # Bytes 21-32: Three physics integers (big-endian uint32)
    # ═══════════════════════════════════════════════════════════
    physics_int_1 = 0  # Game limit
    physics_int_2 = 0
    physics_int_3 = 0

    data.extend(struct.pack('>I', physics_int_1))
    data.extend(struct.pack('>I', physics_int_2))
    data.extend(struct.pack('>I', physics_int_3))
    # Current size: 21 + 12 = 33 bytes

    # ═══════════════════════════════════════════════════════════
    # Bytes 33-36: Gravity constant (fixed-point)
    # ═══════════════════════════════════════════════════════════
    gravity = 20.0  # Standard game gravity
    data.extend(encode_fixed_point(gravity))
    # Current size: 33 + 4 = 37 bytes

    # ═══════════════════════════════════════════════════════════
    # Bytes 37-44: Two physics integers
    # ═══════════════════════════════════════════════════════════
    sky_height = 1000  # Map vertical bound
    physics_int_5 = 0

    data.extend(struct.pack('>I', sky_height))
    data.extend(struct.pack('>I', physics_int_5))
    # Current size: 37 + 8 = 45 bytes

    # ═══════════════════════════════════════════════════════════
    # Bytes 45-48: Physics threshold (fixed-point)
    # ═══════════════════════════════════════════════════════════
    physics_threshold = 1.0
    data.extend(encode_fixed_point(physics_threshold))
    # Current size: 45 + 4 = 49 bytes

    # ═══════════════════════════════════════════════════════════
    # Bytes 49-92: Eleven render floats (fixed-point)
    # ═══════════════════════════════════════════════════════════
    # These control camera/view parameters - use 1.0 as safe defaults
    render_floats = [1.0] * 11
    for value in render_floats:
        data.extend(encode_fixed_point(value))
    # Current size: 49 + 44 = 93 bytes

    # ═══════════════════════════════════════════════════════════
    # Bytes 93-94: Two game mode flags
    # ═══════════════════════════════════════════════════════════
    friendly_fire = False  # Disable friendly fire
    final_flag = False     # Normal mode

    data.append(0x01 if friendly_fire else 0x00)
    data.append(0x01 if final_flag else 0x00)
    # Final size: 93 + 2 = 95 bytes

    assert len(data) == 95, f"Fixed header should be 95 bytes, got {len(data)}"
    return bytes(data)

    Key Points About Fixed Header¶

    1. Exact size: Always exactly 95 bytes, no more, no less
    2. Order matters: Values must be in the exact order shown
    3. Fixed-point encoding: All doubles and floats use value × 65536 encoding (4 bytes each)
    4. Booleans: 1 byte each, 0x00 = false, any non-zero = true
    5. Big-endian integers: All uint32 values use big-endian byte order
    6. Round time default: If negative, client defaults to 60 minutes
    7. Gravity: Affects projectile physics (multiplied by ~-0.85 for vertical motion)
    8. Friendly fire: When 0x00, same-team damage is prevented

    Common Mistakes¶

    Mistake Symptom Fix
    Wrong byte count Client crashes or hangs Verify exactly 95 bytes
    Little-endian encoding Garbled values, weird physics Use > prefix in struct.pack
    Missing fixed-point conversion Physics values are tiny/huge Multiply by 65536 before sending
    Wrong field order Unpredictable behavior Follow the exact order from decompile
    Using 8-byte doubles Extra bytes corrupt next section Both doubles and floats are 4 bytes on wire

    What Comes After the Fixed Header?¶

    After the 95-byte fixed header, the client immediately reads Section 2 (Entity Weapon Data). This is where things get more complex because the size depends on how many entities are registered.

    In Part 4-2, we'll tackle the Entity Weapon Data section - the largest variable-size section of the BEHAVIOR packet.

    Summary: Fixed Header Checklist¶

    Requirement Value
    Size Exactly 95 bytes
    Main flag 1 byte (bool) at offset 0
    Physics doubles 7 × 4 bytes (fixed-point) at offsets 1, 5, 9, 13, 17, 33, 45
    Physics integers 5 × 4 bytes (uint32 BE) at offsets 21, 25, 29, 37, 41
    Render floats 11 × 4 bytes (fixed-point) at offsets 49-92
    Flag booleans 2 × 1 byte at offsets 93, 94
    Byte order All multi-byte values are BIG-ENDIAN
    Fixed-point formula wire_value = int(float_value × 65536)
