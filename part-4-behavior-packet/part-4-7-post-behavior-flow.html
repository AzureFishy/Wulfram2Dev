    Wulfram 2 Server Implementation Guide¶

    Part 4-7: Post-BEHAVIOR Flow Overview¶

    Overview¶

    After successfully processing the BEHAVIOR packet, the client has received all game configuration data but is not yet in the game. This document provides a high-level overview of the packet flow that transitions the client from post-BEHAVIOR through team selection and into active gameplay.

    BEHAVIOR Packet Complete
         ║
         ║ Client has: weapon data, entity stats, vehicle types,
         ║             physics tables, vehicle model parameters
         ║
         ║ Client needs: team info, player assignment, world state
         ║
         ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                   POST-BEHAVIOR PACKET FLOW                             │
│                                                                         │
│  PHASE 2B: TEAM SELECTION SETUP                                         │
│  ─────────────────────────────────                                      │
│  Server ──► TEAM_INFO (0x28)                                           │
│             Client now shows team selection screen                      │
│                                                                         │
│  PHASE 3: GAME ENTRY                                                    │
│  ───────────────────                                                    │
│  Client ──► REINCARNATE (0x25)   Player chooses team                   │
│  Server ──► REINCARNATE (0x25)   Server confirms/responds              │
│  Server ──► PLAYER (0x17)        Assigns player to entity    CRITICAL! │
│  Server ──► GAME_CLOCK (0x2F)    Synchronizes game time                │
│  Server ──► ADD_TO_ROSTER (0x1A) Adds player to team roster            │
│                                                                         │
│  PHASE 4: ACTIVE GAMEPLAY                                               │
│  ────────────────────────                                               │
│  Client ──► WANT_UPDATES (0x39)  Requests game state updates           │
│  Server ──► UPDATE_ARRAY (0x0E)  Sends entity positions/states         │
│  Server ──► WORLD_STATS (0x16)   Map/world information                 │
│             ... bidirectional game traffic ...                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

    Packet Type Quick Reference¶

    Packet Name Type ID Direction Purpose
    TEAM_INFO 0x28 Server→Client Team names/colors for selection screen
    REINCARNATE 0x25 Bidirectional Team/spawn selection and responses
    PLAYER 0x17 Server→Client CRITICAL - Assigns player to entity
    WORLD_STATS 0x16 Server→Client Map name and world parameters
    GAME_CLOCK 0x2F Server→Client Game time synchronization
    ADD_TO_ROSTER 0x1A Server→Client Adds player to team roster
    REMOVE_FROM_ROSTER 0x1B Server→Client Removes player from roster
    WANT_UPDATES 0x39 Client→Server Client requests state updates
    UPDATE_ARRAY 0x0E Server→Client Entity state updates (positions, etc.)
    BIRTH_NOTICE 0x1E Server→Client Entity spawned notification
    DEATH_NOTICE 0x1D Server→Client Entity destroyed notification

    Phase 2B: Team Selection Setup¶

    After BEHAVIOR, the server must send TEAM_INFO to display the team selection screen.

    SERVER                                     CLIENT
   │                                          │
   │  (BEHAVIOR processed successfully)       │
   │                                          │
   │───── TEAM_INFO (0x28) ──────────────────►│
   │      - Team 1 info (ID, names, colors)   │
   │      - Team 2 info (ID, names, colors)   │
   │                                          │
   │                                          │ Client shows
   │                                          │ TEAM SELECTION
   │                                          │ screen
   │                                          │

    Phase 3: Game Entry¶

    When the player selects a team, the client sends REINCARNATE. The server must respond with several packets to complete game entry.

    SERVER                                     CLIENT
   │                                          │
   │                                          │ Player clicks
   │                                          │ "Join Red Team"
   │                                          │
   │◄──── REINCARNATE (0x25) ─────────────────│
   │      - Sub-type (team choice)            │
   │      - Optional message                  │
   │                                          │
   │───── REINCARNATE (0x25) ─────────────────►│
   │      - Sub-type (confirmation/status)    │
   │      - Status message                    │
   │                                          │
   │───── PLAYER (0x17) ─────────────────────►│  ◄── CRITICAL!
   │      - Entity ID (player's tank)         │
   │      - Active flag                       │
   │                                          │
   │───── GAME_CLOCK (0x2F) ─────────────────►│
   │      - Game time sync                    │
   │                                          │
   │───── ADD_TO_ROSTER (0x1A) ──────────────►│
   │      - Player added to team roster       │
   │                                          │
   │                                          │ Client enters
   │                                          │ GAMEPLAY state
   │                                          │

    REINCARNATE Packet (0x25)¶

    The REINCARNATE packet is bidirectional and handles multiple functions: - Client → Server: Team selection, spawn requests - Server → Client: Confirmations, status messages, team changes

    From FUN_0046e280 (lines 78119-78219):

    void FUN_0046e280(param_1, param_2, param_3) {
    FUN_00509db0("REINCARNATE", param_3);
    FUN_005092e0(local_f);       // 1 byte: Sub-type
    FUN_00509340(&local_c);      // String: Message (optional)
    FUN_00509dc0(param_3, param_2);

    switch(local_f[0]) {
        case 0x02: iVar4 = 0x11; break;  // Team assignment
        case 0x04: iVar4 = 0x11; break;  // Spawn type 1
        case 0x05: iVar4 = 0x11; break;  // Spawn type 2
        // ... more sub-types ...
        case 0x0B: iVar4 = 6; break;     // Status change
        case 0x0C: iVar4 = 0xf; break;   // Another status
        // ...
    }
    // Process based on sub-type
}

    REINCARNATE Wire Format:
┌─────────┬──────────────────────────────────────────────────────────────┐
│ Offset  │ Content                                                      │
├─────────┼──────────────────────────────────────────────────────────────┤
│ 0       │ Sub-type (1 byte)                                           │
│ 1       │ Message (length-prefixed string, can be empty)              │
└─────────┴──────────────────────────────────────────────────────────────┘

    REINCARNATE Sub-Types¶

    Sub-Type Hex Direction Description
    0 0x00 C→S Initial/default
    1 0x01 C→S Join team 1
    2 0x02 S→C Team assignment response
    3 0x03 C→S Join team 2
    4-10 0x04-0x0A S→C Various spawn/status responses
    11 0x0B S→C Status change
    12 0x0C S→C Another status
    13 0x0D S→C Conditional status
    14 0x0E S→C Team-specific status
    17 0x11 S→C Generic status
    18 0x12 S→C Message display

    PLAYER Packet (0x17) - CRITICAL¶

    This is the most important packet for game entry! Without it, the client cannot transition from team selection to gameplay.

    From FUN_0046d1f0 (lines 77607-77624):

    void FUN_0046d1f0(param_1, param_2, param_3) {
    FUN_00509db0("PLAYER", param_3);
    FUN_00509310(&local_4);      // 4 bytes: Entity ID (player's vehicle)
    FUN_005092e0(&local_5);      // 1 byte:  Active flag
    FUN_00509dc0(param_3, param_2);

    // Set player state
    DAT_005b8393 = local_5 != '\0';  // Player active
    DAT_005b83e4 = local_4;          // Player's entity ID

    FUN_00443a10();  // Initialize player systems
    FUN_00417ae0();  // Transition to gameplay state
}

    PLAYER Wire Format:
┌─────────┬────────────────────────────────────────────────────┐
│ Offset  │ Content                                            │
├─────────┼────────────────────────────────────────────────────┤
│ 0-3     │ Entity ID (4 bytes, uint32 big-endian)            │
│ 4       │ Active flag (1 byte, 0x00=inactive, 0x01=active)  │
└─────────┴────────────────────────────────────────────────────┘
Total: 5 bytes payload

    Example: 

    PLAYER packet assigning entity ID 12345 as active:
  17 00 05 00 00 30 39 01
  │  └──┬──┘ └────┬────┘ │
  │     │         │      └─ Active = true
  │     │         └─ Entity ID = 12345 (0x3039)
  │     └─ Length = 5 bytes
  └─ Type = 0x17 (PLAYER)

    WORLD_STATS Packet (0x16)¶

    Provides world/map information to the client.

    From FUN_0046cf60 (lines 77500-77523):

    void FUN_0046cf60(param_1, param_2, param_3) {
    FUN_00509db0("WORLD_STATS", param_3);
    FUN_00509340(&local_c);      // String: Map name
    FUN_005092e0(&local_d);      // 1 byte:  Unknown
    FUN_005092e0(&local_e);      // 1 byte:  World type/mode
    FUN_00509310(&local_8);      // 4 bytes: Scale factor (fixed-point)
    FUN_00509dc0(param_3, param_2);

    // After processing, client sends WANT_UPDATES
    iVar1 = FUN_00509cf0("WANT_UPDATES", 0x39, 0, 1);
    if (iVar1 != 0) {
        thunk_FUN_00509520(1);   // Write: 4 bytes (value 1)
        FUN_0050a340(iVar1);     // Send packet
    }
}

    Important: After receiving WORLD_STATS, the client automatically sends WANT_UPDATES!

    WORLD_STATS Wire Format:
┌─────────┬────────────────────────────────────────────────────┐
│ Offset  │ Content                                            │
├─────────┼────────────────────────────────────────────────────┤
│ 0       │ Map name (length-prefixed string)                 │
│ varies  │ Unknown byte (1 byte)                             │
│ varies  │ World type/mode (1 byte)                          │
│ varies  │ Scale factor (4 bytes, fixed-point)               │
└─────────┴────────────────────────────────────────────────────┘

    WANT_UPDATES Packet (0x39)¶

    Client request for game state updates. Sent automatically after WORLD_STATS.

    WANT_UPDATES Wire Format:
┌─────────┬────────────────────────────────────────────────────┐
│ Offset  │ Content                                            │
├─────────┼────────────────────────────────────────────────────┤
│ 0-3     │ Update flags (4 bytes, uint32 big-endian)         │
└─────────┴────────────────────────────────────────────────────┘
Total: 4 bytes payload

    Typically sent with value 1 to request all updates.

    Common Issues and Solutions¶

    Issue: Client Stuck at Team Select¶

    Symptom: Team selection screen shows but clicking a team does nothing.

    Causes: 1. Server not receiving REINCARNATE from client 2. Server not responding with PLAYER packet

    Solution: - Ensure you're listening for REINCARNATE (0x25) packets - Send PLAYER packet after team selection confirmation

    Issue: Client Shows Team Select But Can't Progress¶

    Symptom: Team selection works but game never starts.

    Causes: 1. Missing PLAYER packet (most common!) 2. Entity ID in PLAYER packet is invalid 3. Active flag set to 0

    Solution: 

    # Send PLAYER packet with valid entity ID
entity_id = 12345  # Must be valid entity in your game state
active = True
player_packet = make_packet(0x17, struct.pack('>I', entity_id) + bytes([0x01 if active else 0x00]))
send(player_packet)

    Issue: Player Joins But Can't Move/Act¶

    Symptom: Client enters game but controls don't work.

    Causes: 1. BEHAVIOR packet was malformed (especially Section 5) 2. Entity ID doesn't exist in world state 3. UPDATE_ARRAY not being sent

    Solution: - Verify BEHAVIOR packet is complete (3,115+ bytes) - Ensure entity exists before assigning via PLAYER - Start sending UPDATE_ARRAY packets after WANT_UPDATES

    Issue: Other Players Don't See New Player¶

    Symptom: Player joins but isn't visible to others.

    Causes: 1. ADD_TO_ROSTER not sent to other clients 2. BIRTH_NOTICE not broadcast

    Solution: - Send ADD_TO_ROSTER to all connected clients - Broadcast BIRTH_NOTICE for the new player's entity

    Issue: Player Not Shown in Team Roster¶

    Symptom: Player joins but doesn't appear in scoreboard/roster.

    Causes: 1. ADD_TO_ROSTER not sent, or sent with wrong format 2. Missing required fields (entity_id, team_id, etc.)

    Solution: 

    # Send ADD_TO_ROSTER with all required fields
roster = create_add_to_roster_packet(
    player_id=1,
    entity_id=12345,
    team_id=1,        # Must match the team they joined!
    player_name="Player1"
)
conn.send(roster)

    Note: See Part 6 for the complete create_add_to_roster_packet() implementation.

    Packet Order Summary¶

    Minimum required sequence for game entry:

    Step Packet Direction Required?
    1 TEAM_INFO S→C Yes
    2 REINCARNATE C→S Yes (player action)
    3 REINCARNATE S→C Recommended
    4 PLAYER S→C YES - CRITICAL
    5 GAME_CLOCK S→C Recommended
    6 ADD_TO_ROSTER S→C Recommended
    7 WORLD_STATS S→C Yes (triggers WANT_UPDATES)
    8 WANT_UPDATES C→S Auto (from client)
    9 UPDATE_ARRAY S→C Yes (for gameplay)
