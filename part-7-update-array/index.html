    Wulfram 2 Server Implementation Guide¶

    Part 7: UPDATE_ARRAY & Game State¶

    Overview¶

    Once the client enters gameplay (after receiving PLAYER and WORLD_STATS), the server must continuously send UPDATE_ARRAY (0x0E) packets to update entity positions, states, and other dynamic game data. This is the primary packet for maintaining game state synchronization between server and all connected clients.

    Position in Connection Flow:
┌─────────────────────────────────────────────────────────────────────────┐
│  PHASE 3: GAME ENTRY                                                    │
│  ───────────────────                                                    │
│  ... PLAYER, GAME_CLOCK, WORLD_STATS sent ...                          │
│  Client sends WANT_UPDATES (0x39) automatically                         │
├─────────────────────────────────────────────────────────────────────────┤
│  PHASE 4: ACTIVE GAMEPLAY                                ◄ YOU ARE      │
│  ────────────────────────                                    HERE       │
│  Server ──► UPDATE_ARRAY (0x0E)              Entity state updates       │
│  Server ──► BIRTH_NOTICE (0x1E)              New entity spawns          │
│  Server ──► DELETE_OBJECT (0x15)             Entity removals            │
│  Server ──► TRANSIENT_ARRAY (0x0D)           Projectiles/effects        │
│  Client ──► ACTION_UPDATE (0x0A)             Player inputs              │
│             ... continuous bidirectional traffic ...                    │
└─────────────────────────────────────────────────────────────────────────┘

    Critical Concept: Bit-Stream Format¶

    The UPDATE_ARRAY packet uses a bit-stream format where field sizes are configured by the BEHAVIOR packet. This means:

    1. Field widths are NOT fixed - they depend on values sent in BEHAVIOR
    2. Data is packed at the bit level, not byte level
    3. The client uses DAT_00678134 (BEHAVIOR config) to know how to parse

    This is why BEHAVIOR must be sent correctly before gameplay!

    // ═══════════════════════════════════════════════════════════
    // HEADER: Read timestamp and entity count
    // ═══════════════════════════════════════════════════════════
    FUN_00509310();                    // Read 4 bytes: Timestamp
    _DAT_0067d0c4 = local_d0;          // Store timestamp

    FUN_00509320(8, &local_c4);        // Read 8 bits: Entity count

    // ═══════════════════════════════════════════════════════════
    // LOOP: Process each entity update
    // ═══════════════════════════════════════════════════════════
    for (int i = 0; i < entity_count; i++) {
        // Read entity ID using BEHAVIOR-configured bit width
        FUN_00509310();                // 4 bytes: Entity ID

        // Read presence flags (1 bit each, packed)
        FUN_00509320(1, &local_a0);    // Has extended data?
        FUN_00509320(10, &local_dc);   // 10 presence flags

        if (flags & 0x001) {  // Bit 0: Full entity data
            // Entity type, team, spawn info
            FUN_00509320(*DAT_00678134, &entity_type);
            FUN_00509320(DAT_00678134[0x20], &team_id);
            FUN_00509320(DAT_00678134[0x30], &spawn_param1);
            FUN_00509320(DAT_00678134[0x30], &spawn_param2);
        }

        if (flags & 0x002) {  // Bit 1: Position X
            FUN_0047cfa0(...);  // Read position component
        }

        if (flags & 0x004) {  // Bit 2: Position Y
            FUN_0047cfa0(...);  // Read position component
        }

        if (flags & 0x008) {  // Bit 3: Position Z
            FUN_0047cfa0(...);  // Read position component
        }

        if (flags & 0x010) {  // Bit 4: Velocity/Orientation set 1
            FUN_0047cfa0(...);
        }

        if (flags & 0x020) {  // Bit 5: Speed/movement
            FUN_00509320(DAT_00678134[0x50], &speed);
        }

        if (flags & 0x040) {  // Bit 6: Weapon state
            FUN_0047d110(param_3);  // Read weapon data
        }

        if (flags & 0x080) {  // Bit 7: Health/damage
            FUN_00509320(DAT_00678134[0x80], &health);
        }

        if (flags & 0x100) {  // Bit 8: Target/aim
            FUN_00509310();  // Read target data
        }

        if (flags & 0x200) {  // Bit 9: Extended state
            // Additional state data
        }

        // Apply updates to entity

    Entity Update Structure (variable length, bit-packed):
┌─────────────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────────────┤

├─────────────────────────────────────────────────────────────────────────┤

    Presence Flags (10 bits)¶

    Bit Mask Field Description
    0 0x001 Full Data Entity type, team, spawn params
    1 0x002 Position X X coordinate data
    2 0x004 Position Y Y coordinate data
    3 0x008 Position Z Z coordinate data
    4 0x010 Velocity Velocity/orientation set
    5 0x020 Speed Movement speed
    6 0x040 Weapon Weapon state data
    7 0x080 Health Health/damage value
    8 0x100 Target Targeting data
    9 0x200 Extended Additional state

    Section 5: Implementation Strategy¶

    Simplified UPDATE_ARRAY for Testing¶

    For initial testing, you can create a simplified UPDATE_ARRAY that sends minimal data:

    import struct
import io

class BitWriter:
    """Write data at the bit level."""

    def __init__(self):
        self.buffer = bytearray()
        self.current_byte = 0
        self.bit_position = 0

    def write_bits(self, value, num_bits):
        """Write a value using the specified number of bits."""
        for i in range(num_bits - 1, -1, -1):
            bit = (value >> i) & 1
            self.current_byte = (self.current_byte << 1) | bit
            self.bit_position += 1

            if self.bit_position == 8:
                self.buffer.append(self.current_byte)
                self.current_byte = 0
                self.bit_position = 0

    def write_uint32(self, value):
        """Write a 32-bit value (byte-aligned)."""
        self.flush()  # Align to byte boundary first
        self.buffer.extend(struct.pack('>I', value))

    def write_fixed_point(self, float_value, num_bits):
        """Write a fixed-point value."""
        max_val = (1 << num_bits) - 1
        # Scale float to integer range
        int_val = int(float_value * 65536) & max_val
        self.write_bits(int_val, num_bits)

    def flush(self):
        """Pad remaining bits to byte boundary."""
        if self.bit_position > 0:
            self.current_byte <<= (8 - self.bit_position)
            self.buffer.append(self.current_byte)
            self.current_byte = 0
            self.bit_position = 0

    def get_bytes(self):
        """Get the final byte array."""
        self.flush()
        return bytes(self.buffer)


def create_update_array_simple(timestamp, entities):
    """
    Create a simplified UPDATE_ARRAY packet.

    Args:
        timestamp: Current game timestamp
        entities: List of (entity_id, x, y, z) tuples

    Returns:
        Complete packet bytes
    """
    writer = BitWriter()

    # Header
    writer.write_uint32(timestamp)
    writer.write_bits(len(entities), 8)  # Entity count

    for entity_id, x, y, z in entities:
        # Entity ID (32 bits)
        writer.write_uint32(entity_id)

        # Extended flag (1 bit) - not extended
        writer.write_bits(0, 1)

        # Presence flags (10 bits) - only position bits set (0x00E)
        # Bits 1, 2, 3 = X, Y, Z position
        writer.write_bits(0x00E, 10)

        # Position data (simplified - using fixed bit widths)
        # In real implementation, use BEHAVIOR-configured widths
        writer.write_fixed_point(x, 16)
        writer.write_fixed_point(y, 16)
        writer.write_fixed_point(z, 16)

    payload = writer.get_bytes()

    # Create packet frame
    return bytes([0x0E]) + struct.pack('>H', len(payload)) + payload

    Full Implementation Notes¶

    For a complete implementation:

    1. Track BEHAVIOR configuration: Store the bit widths from BEHAVIOR packet
    2. Entity management: Maintain a server-side entity list with positions, states
    3. Delta compression: Only send fields that changed (using presence flags)
    4. Tick rate: Send UPDATE_ARRAY at regular intervals (e.g., 20-60 Hz)
    class GameState:
    def __init__(self):
        self.entities = {}  # entity_id -> EntityState
        self.timestamp = 0
        self.behavior_config = None  # From BEHAVIOR packet

    def create_update_packet(self):
        """Create UPDATE_ARRAY with all active entities."""
        self.timestamp += 1

        # Build entity list
        entity_updates = []
        for entity_id, state in self.entities.items():
            if state.active:
                entity_updates.append((
                    entity_id,
                    state.x, state.y, state.z
                ))

        return create_update_array_simple(self.timestamp, entity_updates)

    Basic Game Loop Structure¶

    import time
import threading

class Wulfram2Server:
    def __init__(self):
        self.game_state = GameState()
        self.clients = []
        self.running = False
        self.tick_rate = 20  # Updates per second

    def start_game_loop(self):
        """Start the game update loop."""
        self.running = True

        tick_time = 1.0 / self.tick_rate

        while self.running:
            start = time.time()

            # Update game state
            self.update_game_state()

            # Send updates to all clients
            update_packet = self.game_state.create_update_packet()
            for client in self.clients:
                if client.in_game:
                    client.send(update_packet)

            # Sleep to maintain tick rate
            elapsed = time.time() - start
            if elapsed < tick_time:
                time.sleep(tick_time - elapsed)

            entity.z += entity.vz * (1.0 / self.tick_rate)

    Minimal Gameplay Server¶

    import struct
import socket
import threading
import time

class MinimalGameServer:
    """
    Minimal server that gets a client into gameplay with UPDATE_ARRAY.
    """

    def __init__(self):
        self.entities = {
            1: {'x': 100.0, 'y': 0.0, 'z': 100.0},  # Player entity
            2: {'x': 200.0, 'y': 0.0, 'z': 200.0},  # Another entity
        }
        self.timestamp = 0

    def create_update_array(self):
        """Create a simple UPDATE_ARRAY packet."""
        self.timestamp += 1

        # Simple fixed-width implementation for testing
        payload = bytearray()

        # Timestamp (4 bytes)
        payload.extend(struct.pack('>I', self.timestamp))

        # Entity count (1 byte for simplicity)
        payload.append(len(self.entities))

        # For each entity, send a minimal update
        for entity_id, state in self.entities.items():
            # Entity ID (4 bytes)
            payload.extend(struct.pack('>I', entity_id))

            # Simplified presence flags (2 bytes) - position only
            payload.extend(struct.pack('>H', 0x000E))

            # Position as fixed-point (4 bytes each)
            payload.extend(struct.pack('>i', int(state['x'] * 65536)))
            payload.extend(struct.pack('>i', int(state['y'] * 65536)))
            payload.extend(struct.pack('>i', int(state['z'] * 65536)))

        return bytes([0x0E]) + struct.pack('>H', len(payload)) + payload

    def game_loop(self, conn):
        """Send updates at 10 Hz."""
        while True:
            # Update entity positions (simple circular motion for testing)
            import math
            t = self.timestamp * 0.1
            self.entities[2]['x'] = 200.0 + math.cos(t) * 50
            self.entities[2]['z'] = 200.0 + math.sin(t) * 50

            # Send update
            update = self.create_update_array()
            try:
                conn.send(update)
            except:
                break

            time.sleep(0.1)  # 10 Hz

    Troubleshooting¶

    Problem: Client Shows Wrong Positions¶

    Cause: Bit widths don't match BEHAVIOR configuration.

    Solution: Use the exact bit widths configured in your BEHAVIOR packet: 

    # Store these from your BEHAVIOR packet
entity_type_bits = DAT_00678134[0x00]
team_bits = DAT_00678134[0x20]
# ... etc

    Problem: Client Crashes During UPDATE_ARRAY¶

    Cause: Malformed bit-stream or wrong entity count.

    Solution: - Verify entity count matches actual entities sent - Ensure all presence flag bits have corresponding data - Check bit alignment at packet end

    Problem: Entities Don't Move¶

    Cause: UPDATE_ARRAY not being sent, or wrong entity IDs.

    Solution: - Verify server is calling game loop - Check entity IDs match those assigned via PLAYER packet - Confirm presence flags include position bits (0x00E)

    Problem: Client Disconnects After First UPDATE¶

    Cause: Server not handling client packets (WANT_UPDATES, ACTION_UPDATE).

    Solution: Implement handlers for all client packet types during gameplay.

    Common Mistakes¶

    Mistake Symptom Fix
    Wrong bit widths Garbled positions Match BEHAVIOR config exactly
    Missing entities Player invisible Include player entity in updates
    No game loop Frozen game Start update thread after WANT_UPDATES
    Bad timestamp Time desync Increment timestamp each update
    Missing BIRTH_NOTICE Entities don't appear Send BIRTH_NOTICE for new entities

    Verification Checklist¶

    Before considering gameplay working:

    • [ ] Server handles WANT_UPDATES from client
    • [ ] Server starts game loop after WANT_UPDATES
    • [ ] UPDATE_ARRAY sent at consistent rate
    • [ ] Player's entity included in updates
    • [ ] Entity positions update over time
    • [ ] Client can see themselves in game world
    • [ ] BIRTH_NOTICE sent for new entities
    • [ ] DELETE_OBJECT sent when entities removed

    Quick Reference: Gameplay Packets¶

    Packet Type Direction Purpose
    UPDATE_ARRAY 0x0E S→C Entity state updates
    TRANSIENT_ARRAY 0x0D S→C Projectiles/effects
    BIRTH_NOTICE 0x1E S→C Entity spawned
    DELETE_OBJECT 0x15 S→C Entity removed
    ACTION_UPDATE 0x0A C→S Player input
    WANT_UPDATES 0x39 C→S Request updates (auto)

    Implementation Priority¶

    For a minimal working game:

    1. UPDATE_ARRAY (0x0E) - REQUIRED - Send entity positions
    2. Handle WANT_UPDATES (0x39) - Trigger game loop start
    3. BIRTH_NOTICE (0x1E) - Required for new entity visibility
    4. DELETE_OBJECT (0x15) - Required for entity cleanup

    Optional but recommended: - TRANSIENT_ARRAY (0x0D) - For projectiles - ACTION_UPDATE handling (0x0A) - For player control
