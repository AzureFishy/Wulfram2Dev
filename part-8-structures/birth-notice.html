    BIRTH_NOTICE Packet (0x1E) - Complete Structure Reference¶

    Overview¶

    The BIRTH_NOTICE packet (0x1E) notifies the client that an entity has "spawned" or become active in the game world. When received, the client marks the entity as "born" and associates it with an owner (the player controlling it).

    This is a simple but important packet that activates entities after they've been created.

    Purpose:
┌─────────────────────────────────────────────────────────────────────────┐
│  BIRTH_NOTICE tells the client:                                         │
│                                                                         │
│  "Entity X is now active and controlled by Entity Y"                    │
│                                                                         │
│  • Sets the entity's "born" flag to TRUE                                │
│  • Links the entity to its owner/controller                             │
└─────────────────────────────────────────────────────────────────────────┘

    Position in Connection Flow¶

    Position in Connection Flow:
┌─────────────────────────────────────────────────────────────────────────┐
│  PHASE 3: GAME ENTRY                                                    │
│  ───────────────────                                                    │
│  Client ──► REINCARNATE (0x25)               Player chooses team        │
│  Server ──► REINCARNATE (0x25)               Server confirms            │
│  Server ──► PLAYER (0x17)                    Assigns player entity      │
│  Server ──► GAME_CLOCK (0x2F)                Time synchronization       │
│  Server ──► ADD_TO_ROSTER (0x1A)             Adds to team roster        │
├─────────────────────────────────────────────────────────────────────────┤
│  Server ──► BIRTH_NOTICE (0x1E)              ◄ YOU ARE HERE             │
│             Activates the player's vehicle entity                       │
├─────────────────────────────────────────────────────────────────────────┤
│  Server ──► WORLD_STATS (0x16)               Map info                   │
│  Client ──► WANT_UPDATES (0x39)              Auto-sent by client        │
└─────────────────────────────────────────────────────────────────────────┘

    Packet Registration¶

    From the client initialization code:

    // From line 77046
FUN_00509b40(0x1e, FUN_0046d880);    // Register type 0x1E → BIRTH_NOTICE handler

    The packet name is registered for debugging:

    // From line 94149
FUN_0044a020("BIRTH_NOTICE", 0x1e);

    The Handler Code¶

    From FUN_0046d880 (lines 77796-77820):

    void FUN_0046d880(undefined4 param_1, undefined4 param_2, undefined4 param_3)
{
    int iVar1;
    int *piVar2;
    undefined4 local_8;           // Owner Entity ID
    int local_4;                  // Entity ID

    FUN_00509db0("BIRTH_NOTICE", param_3);    // Set packet name for errors

    // ═══════════════════════════════════════════════════════════
    // READ PACKET DATA
    // ═══════════════════════════════════════════════════════════
    FUN_00509310(&local_4);       // 4 bytes: Entity ID (uint32 BE)
    FUN_00509310(&local_8);       // 4 bytes: Owner Entity ID (uint32 BE)
    FUN_00509dc0(param_3, param_2);           // Byte alignment

    // ═══════════════════════════════════════════════════════════
    // FIND ENTITY AND UPDATE FLAGS
    // ═══════════════════════════════════════════════════════════
    piVar2 = *(int **)(DAT_00678390 + 8);     // Get entity list head

    if (piVar2 != (int *)0x0) {
        // Search for entity with matching ID
        while (iVar1 = piVar2[2], *(int *)(iVar1 + 0xc) != local_4) {
            piVar2 = (int *)*piVar2;          // Move to next entity
            if (piVar2 == (int *)0x0) {
                return;                        // Entity not found
            }
        }

        // Entity found - update its state
        *(undefined4 *)(iVar1 + 0x3c) = 1;    // Set "born" flag = TRUE
        *(undefined4 *)(iVar1 + 0x40) = local_8;  // Set owner entity ID
    }
    return;
}

    What The Handler Does¶

    The handler performs these steps:

    1. Read Entity ID (4 bytes) - The entity being "born"
    2. Read Owner Entity ID (4 bytes) - The entity that controls/owns it
    3. Search Entity List - Find the entity with matching ID
    4. Set Born Flag - Mark entity as active (entity + 0x3c = 1)
    5. Set Owner - Link to controlling entity (entity + 0x40 = owner_id)
    Entity Structure (relevant offsets):
┌─────────┬──────────────────────────────────────────────────────────────┐
│ Offset  │ Description                                                  │
├─────────┼──────────────────────────────────────────────────────────────┤
│ 0x0C    │ Entity ID (used to find entity in list)                      │
│ 0x3C    │ Born/Active Flag (0 = not born, 1 = born/active)             │
│ 0x40    │ Owner Entity ID (who controls this entity)                   │
└─────────┴──────────────────────────────────────────────────────────────┘

    Payload Structure¶

    BIRTH_NOTICE Payload Layout:
┌─────────┬──────┬────────────────┬───────────────────────────────────────────┐
│ Offset  │ Size │ Type           │ Description                               │
├─────────┼──────┼────────────────┼───────────────────────────────────────────┤
│ 0       │ 4    │ uint32 BE      │ Entity ID                                 │
│ 4       │ 4    │ uint32 BE      │ Owner Entity ID                           │
└─────────┴──────┴────────────────┴───────────────────────────────────────────┘

Total Payload Size: 8 bytes (fixed)

    Field Details¶

    Field 1: Entity ID (4 bytes)¶

    Offset: 0
Size:   4 bytes
Type:   uint32 big-endian
Reader: FUN_00509310

Description:
  The ID of the entity that is being "born" or activated.

  This is typically the player's vehicle (Tank, Scout, etc.) that was
  assigned via the PLAYER packet.

  ★ IMPORTANT: This entity must already exist in the client's entity list! ★

  The BIRTH_NOTICE doesn't create the entity - it activates an existing one.
  The entity should have been created via UPDATE_ARRAY or similar packets.

Example:
  Entity ID = 12345 (0x3039)
  Wire bytes: 00 00 30 39

    Field 2: Owner Entity ID (4 bytes)¶

    Offset: 4
Size:   4 bytes
Type:   uint32 big-endian
Reader: FUN_00509310

Description:
  The ID of the entity that "owns" or controls this entity.

  For a player's vehicle:
  - Owner Entity ID = same as Entity ID (player controls themselves)

  For projectiles, drones, or spawned objects:
  - Owner Entity ID = the player entity that created/fired it

  This creates a parent-child relationship:
  - Projectile → owned by → Player who fired it
  - Drone → owned by → Player who deployed it

Example (player vehicle - self-owned):
  Entity ID = 12345
  Owner Entity ID = 12345  (same - player owns their own vehicle)
  Wire bytes: 00 00 30 39

Example (projectile owned by player):
  Entity ID = 50000 (the projectile)
  Owner Entity ID = 12345 (the player who fired it)
  Wire bytes: 00 00 30 39

    Building BIRTH_NOTICE in Python¶

    import struct

def make_packet(packet_type, payload):
    """Wrap payload in a complete packet frame."""
    return bytes([packet_type]) + struct.pack('>H', len(payload)) + payload

def create_birth_notice(entity_id, owner_entity_id=None):
    """
    Create a BIRTH_NOTICE packet (0x1E).

    Args:
        entity_id:       The entity being "born" / activated (uint32)
        owner_entity_id: The entity that owns/controls this entity (uint32)
                         If None, defaults to entity_id (self-owned)

    Returns:
        Complete BIRTH_NOTICE packet as bytes

    Usage:
        For player vehicles: owner = entity itself (self-owned)
        For projectiles: owner = player who fired it
        For drones/spawns: owner = player who deployed it
    """
    if owner_entity_id is None:
        owner_entity_id = entity_id  # Self-owned (typical for player vehicles)

    payload = struct.pack('>I', entity_id) + struct.pack('>I', owner_entity_id)

    return make_packet(0x1E, payload)

    Example Usage¶

    # Example 1: Player spawns in their vehicle (self-owned)
# The player's entity owns itself
birth_packet = create_birth_notice(
    entity_id=12345,
    owner_entity_id=12345    # Same as entity_id - player controls themselves
)
# Or simply:
birth_packet = create_birth_notice(entity_id=12345)  # owner defaults to self

# Example 2: Player fires a projectile
# The projectile is owned by the player who fired it
projectile_birth = create_birth_notice(
    entity_id=50000,         # The projectile entity
    owner_entity_id=12345    # The player who fired it
)

# Example 3: Player deploys a drone/turret
drone_birth = create_birth_notice(
    entity_id=60000,         # The drone entity
    owner_entity_id=12345    # The player who deployed it
)

    When to Send BIRTH_NOTICE¶

    During Player Spawn Sequence¶

    # Complete spawn sequence after player selects team:

# 1. Confirm team selection
conn.send(create_reincarnate_response(sub_type=0x02, message="Welcome!"))

# 2. Assign player to their entity
conn.send(create_player_packet(entity_id=12345))

# 3. Send game clock
conn.send(create_game_clock())

# 4. Add player to roster
conn.send(create_add_to_roster(
    player_id=1, 
    entity_id=12345, 
    team_id=1, 
    player_name="Baffler"
))

# 5. Activate the player's entity
conn.send(create_birth_notice(entity_id=12345))  # ◄ BIRTH_NOTICE

# 6. Send world info (triggers WANT_UPDATES)
conn.send(create_world_stats())

    When Entities Are Created Mid-Game¶

    # When a player fires a weapon that creates an entity:
projectile_id = allocate_entity_id()

# First: Create the entity via UPDATE_ARRAY (so it exists in client's list)
# Then: Send BIRTH_NOTICE to activate it
conn.send(create_birth_notice(
    entity_id=projectile_id,
    owner_entity_id=player_entity_id
))

    Relationship to Other Packets¶

    Entity Lifecycle:
═══════════════════════════════════════════════════════════════════════════

1. Entity Created (via UPDATE_ARRAY or initial state)
   └─ Entity exists in client's entity list but is NOT active

2. BIRTH_NOTICE sent
   └─ Entity marked as "born" (active)
   └─ Entity linked to owner
   └─ Entity now participates in game logic

3. Entity Destroyed (via DEATH_NOTICE or similar)
   └─ Entity marked as inactive/dead

═══════════════════════════════════════════════════════════════════════════

Packet Relationships:
┌────────────────────┬──────────────────────────────────────────────────────┐
│ Packet             │ Relationship to BIRTH_NOTICE                         │
├────────────────────┼──────────────────────────────────────────────────────┤
│ PLAYER (0x17)      │ Assigns entity to player - BIRTH_NOTICE activates it │
│ ADD_TO_ROSTER      │ Adds player to team list - uses same entity ID       │
│ UPDATE_ARRAY       │ Creates/updates entity data - BIRTH_NOTICE activates │
│ DEATH_NOTICE       │ Opposite of BIRTH_NOTICE - deactivates entity        │
└────────────────────┴──────────────────────────────────────────────────────┘

    Critical Requirements¶

    1. Entity Must Exist Before BIRTH_NOTICE¶

    The BIRTH_NOTICE doesn't CREATE an entity - it ACTIVATES one.

The entity must already exist in the client's entity list before
sending BIRTH_NOTICE. If the entity doesn't exist, the handler
simply returns without doing anything.

Correct Order:
  1. Create entity (via UPDATE_ARRAY or initial game state)
  2. Send BIRTH_NOTICE (activates the entity)

Wrong Order:
  1. Send BIRTH_NOTICE (entity not found - does nothing!)
  2. Create entity (entity exists but not marked as born)

    2. Entity ID Must Match PLAYER Packet¶

    For player vehicles, the entity_id in BIRTH_NOTICE must match:
  - PLAYER packet entity_id
  - ADD_TO_ROSTER entity_id

All three packets should reference the same entity:
  PLAYER:         entity_id = 12345
  ADD_TO_ROSTER:  entity_id = 12345
  BIRTH_NOTICE:   entity_id = 12345  ← Must match!

    3. Owner Entity ID for Different Entity Types¶

    Entity Type      │ Owner Entity ID
─────────────────┼────────────────────────────────────
Player Vehicle   │ Same as Entity ID (self-owned)
Projectile       │ Player who fired it
Drone/Turret     │ Player who deployed it
Building         │ Team or player who built it
NPC/AI           │ Possibly 0 or server-controlled ID

    Common Mistakes¶

    Mistake Symptom Fix
    Entity doesn't exist yet BIRTH_NOTICE has no effect Create entity first via UPDATE_ARRAY
    Wrong entity ID Wrong entity activated Use same ID as PLAYER packet
    Forgot BIRTH_NOTICE entirely Player can't control vehicle Add to spawn sequence
    Wrong owner for projectiles Kill credit goes to wrong player Set owner to firing player
    Sent before entity created Entity not marked as born Send after UPDATE_ARRAY

    Verification Checklist¶

    Before sending BIRTH_NOTICE, verify:

    • [ ] Entity exists in game state (created via UPDATE_ARRAY or similar)
    • [ ] Entity ID matches the PLAYER packet (for player vehicles)
    • [ ] Entity ID matches ADD_TO_ROSTER (for player vehicles)
    • [ ] Owner Entity ID is set correctly:
    • [ ] Self-owned for player vehicles
    • [ ] Parent player for projectiles/spawns
    • [ ] Values are big-endian encoded
    • [ ] Packet is sent AFTER entity creation, not before

    Complete Player Spawn Example¶

    def spawn_player(conn, player_id, entity_id, team_id, player_name):
    """
    Complete sequence to spawn a player after team selection.
    """
    # 1. Confirm team selection
    conn.send(create_reincarnate_response(
        sub_type=0x02, 
        message=f"Welcome to Team {team_id}!"
    ))

    # 2. Assign player to entity
    conn.send(create_player_packet(entity_id=entity_id))

    # 3. Time sync
    conn.send(create_game_clock())

    # 4. Add to roster (visible to all players)
    conn.send(create_add_to_roster(
        player_id=player_id,
        entity_id=entity_id,
        team_id=team_id,
        player_name=player_name
    ))

    # 5. ★ Activate the entity ★
    conn.send(create_birth_notice(
        entity_id=entity_id,
        owner_entity_id=entity_id  # Self-owned
    ))

    # 6. Send world info (triggers client to send WANT_UPDATES)
    conn.send(create_world_stats())

    # Client is now ready to receive UPDATE_ARRAY and play!
