    Wulfram 2 Server Implementation Guide¶

    Part 9: UDP Protocol¶

    Overview¶

    Wulfram 2 uses UDP for low-latency game data that can tolerate occasional packet loss:

    • Voice chat (VOICE packets)
    • Real-time position/state updates (PRIORITY packets)
    • Weapon state synchronization (WEAPONS packets)
    • Graph/telemetry data (GRAPH packets)
    • Connection authentication (D_ Protocol packets)

    Important: UDP packet type codes exist in a separate namespace from TCP. Type 0x13 over TCP is HELLO, but over UDP it's GRAPH_DATA. See Part 1 for the complete comparison table.

    UDP Architecture Overview¶

    Wulfram 2's UDP system has two distinct layers:

    ┌─────────────────────────────────────────────────────────────┐
│                    Game Data Packets                         │
│  VOICE(0x0A), PRIORITY(0x0B), WEAPONS(0x0C), GRAPH(0x13)    │
├─────────────────────────────────────────────────────────────┤
│              D_ Protocol Layer (Type 0x08)                   │
│  D_HANDSHAKE, D_ACK, D_IGNORE, D_SET_START                  │
│  (Connection auth, stream management, sequencing)           │
├─────────────────────────────────────────────────────────────┤
│                     UDP Transport                            │
│  Raw datagrams to/from server UDP port                      │
└─────────────────────────────────────────────────────────────┘

    Game Data Packets (0x09-0x14): Direct payload packets for game state, voice, etc.

    D_ Protocol Layer (0x08): A sub-protocol for connection management. UDP type 0x08 is a carrier frame that encapsulates D_ packets (D_HANDSHAKE, D_ACK, etc.). See Part 9.6: D_HANDSHAKE for complete details.

    UDP Connection Establishment¶

    The UDP channel is established through a multi-step process:

    CLIENT                                              SERVER
   │                                                   │
   │◄──────────── TCP Connection ─────────────────────│
   │                                                   │
   │◄──── TCP: HELLO sub=1 (UDP port info) ───────────│
   │                                                   │
   │     Client binds UDP socket to server port       │
   │                                                   │
   │───── UDP[0x08]: Client sends initial packet ────►│
   │                                                   │
   │◄──── UDP[0x08]: D_HANDSHAKE ─────────────────────│
   │      (D_type=3, sequence, session, streams)      │
   │                                                   │
   │───── UDP[0x08]: D_ACK ──────────────────────────►│
   │      (D_type=2, sub_type=0, sequence)            │
   │                                                   │
   │          UDP Channel Authenticated               │
   │                                                   │
   │◄────────── Game Data: PRIORITY, VOICE ──────────►│
   │                                                   │

    Key Insight: The 0x08 packet type is NOT a simple acknowledgment—it's a carrier frame for the D_ protocol. The first byte of the 0x08 payload determines the actual D_ packet type.

    UDP Packet Frame Structure¶

    UDP packets use a simple framing with an implicit length (from the datagram):

    UDP Packet Structure:
┌──────────┬─────────────────────────────────┐
│  TYPE    │           PAYLOAD               │
│ (1 byte) │         (variable)              │
└──────────┴─────────────────────────────────┘

    For type 0x08 (D_ Protocol), the payload has additional structure:

    D_ Protocol Frame (Type 0x08):
┌──────────┬──────────┬───────────────────────────────┐
│ UDP Type │ D_Type   │      D_ Payload               │
│   0x08   │ (1 byte) │      (variable)               │
└──────────┴──────────┴───────────────────────────────┘

    UDP Packet Type Reference¶

    Source: Packet analysis by baffler, verified against decompiled client code.

    Type Name Direction Description
    0x08 D_ Protocol Bidirectional Carrier for D_HANDSHAKE, D_ACK, D_IGNORE, D_SET_START
    0x09 ROOT C→S Root/base packet
    0x0A VOICE C→S Voice chat audio data
    0x0B PRIORITY S→C Primary game state updates
    0x0C WEAPONS S→C Weapon state synchronization
    0x0D ACK_WPNS S→C Weapon acknowledgment
    0x0E PRIORITY2 S→C Secondary priority updates
    0x10 RESENDS S→C Packet resend requests
    0x11 UNKNOWN ? ⚠️ Unidentified
    0x12 GRAPH_RESPONSE S→C Graph query response
    0x13 GRAPH_DATA S→C Telemetry/graph data
    0x14 IGNORE S→C Ignore instruction

    D_ Protocol Sub-Types (within 0x08)¶

    D_Type Name Purpose
    0x01 D_IGNORE Instruct receiver to ignore a stream
    0x02 D_ACK Acknowledge D_ protocol packets
    0x03 D_HANDSHAKE Stream configuration and authentication
    0x04 D_SET_START Set stream start positions

    Reference: Packet Name Lookup (Python)¶

    Updated packet identification utility:

    def get_udp_packet_name(type_byte):
    """Get the name of a UDP packet by its type byte"""
    names = {
        0x08: "D_ Protocol (D_HANDSHAKE/D_ACK/D_IGNORE/D_SET_START)",
        0x09: "ROOT (UDP)",
        0x0A: "VOICE (UDP)",
        0x0B: "PRIORITY (UDP)",
        0x0C: "WEAPONS (UDP)",
        0x0D: "ACK_WPNS (UDP)",
        0x0E: "PRIORITY2 (UDP)",
        0x10: "RESENDS (UDP)",
        0x11: "UNKNOWN (0x11 UDP)",
        0x12: "GRAPH_RESPONSE (UDP)",
        0x13: "GRAPH_DATA (UDP)",
        0x14: "IGNORE (UDP)",
    }
    return names.get(type_byte, f"UNKNOWN NOT LISTED (0x{type_byte:02X} UDP)")


def get_d_protocol_name(d_type_byte):
    """Get the name of a D_ protocol packet by its type byte"""
    names = {
        0x01: "D_IGNORE",
        0x02: "D_ACK", 
        0x03: "D_HANDSHAKE",
        0x04: "D_SET_START",
    }
    return names.get(d_type_byte, f"D_UNKNOWN (0x{d_type_byte:02X})")


def parse_udp_packet(data):
    """Parse a UDP packet and identify its type"""
    if len(data) < 1:
        return "EMPTY"

    udp_type = data[0]
    name = get_udp_packet_name(udp_type)

    # If it's a D_ protocol packet, decode the sub-type
    if udp_type == 0x08 and len(data) >= 2:
        d_type = data[1]
        d_name = get_d_protocol_name(d_type)
        return f"{name} -> {d_name}"

    return name

    Quick Reference: When to Use Each Packet¶

    Scenario Packet Type Notes
    Initial UDP auth 0x08 → D_HANDSHAKE (0x03) Must complete before other UDP
    Acknowledge D_ packets 0x08 → D_ACK (0x02) Response to D_HANDSHAKE, etc.
    UDP keepalive 0x08 Simple ping when idle
    Voice transmission 0x0A Audio data, C→S
    Game state sync 0x0B, 0x0E Position, health, etc. S→C
    Weapon updates 0x0C, 0x0D Weapon state, S→C
